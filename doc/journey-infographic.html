The Prompt That Started It All

Nadav's kickoff message — the actual first prompt

"I am an Cityhive director. Looking to implement a MySMS Messenger."

"You will create a scrum team in the form of several Claude agents. We will have an architect, who will create an HLD. A tech lead who will write the tech design. Then we will have a testing specialist who will make sure to QA the system, in every deliverable. The QA specialist is also a chaos engineer who will try to break the design/system. A security specialist will look for vulnerabilities. A team lead will also be a scrum master, and will create scrum user stories, and assign them to a developer. We will have one senior developer and one junior, implementing the backend code And a senior frontend implementing the frontend code."

"Each story completed, will be handed to the QA agent. And the developer will go and fetch more work from the lead. Planning (by architect and by tech lead) will be on Opus. Other agents will be running on Sonnet. You will implement the team lead as a workflow, fanning out concurrent tasks, whenever possible to parallelize. Assigning to either one of the developers, based on task complexity."

"Based on the tech design, checkpoints will be defined, each checkpoint will be pushed into git, so that we have multiple small commits, and not just one large at the end. I want the code to be testable. i.e. implement IoC for e.g. the DAL, so that a client can decide to swap to a different DB, with just a config change. Plus, DAL and IoC will yield testable code."

"When code complete, you will do the following loop engineering: system-wide CR → fix all findings → loop till nothing found in CR. The loop engineering will be capped at $300."

"It would be very interesting to see a sequence diagram with interactions between the agents. Create an infographic HTML to show the teamwork messages and collaboration. HTML as well as the HLD and design document will go under /doc in git."

"For any blockers, ping me, so I can decide how to unblock."

Quoted verbatim — this is what actually kicked the whole project off, before a single line of the app existed.

The attached exercise spec — MySMS Messenger (CityHive)

"A full stack web app allowing the end users to send SMS messages as well as displaying them."

  • Frontend in Angular's latest stable version
  • Backend (API) in Ruby on Rails with MongoDB as the database
  • Sending an SMS through the Twilio API, on a free account

Core functionality requested: send a message via the backend API; store sent messages in a DB the backend manages; list previously-sent messages via a listing endpoint; only messages sent by the caller's session should be visible.

Bonus 1 — swap the session-ID scoping for real user accounts (signup/login/logout), using a built-in/well-known auth mechanism, never hand-rolled.
Bonus 2 — deploy the app: a live, fully-functional demo.
Bonus 3 — "Add a reflection to the message cards showing that twilio successfully delivered the message," via Twilio's status-callback webhooks.

"Once done, push your code to a public GitHub repo along with a link to the live demo."

This is the product spec the director attached — the actual requirements handed to the simulated Architect.

The Journey

Core buildsimulated scrum team

Architect (HLD) → Tech Lead (tech design + CP1–CP12 checkpoint plan) → Scrum Master fanning work out to senior/junior backend + senior frontend devs, in parallel where possible. Two IoC seams designed in from day one: a repository interface (Mongo swappable for an in-memory fake) and an SMS-gateway interface (Twilio swappable for a fake) — the seams that later made every bonus a config change, not a rewrite.

QA + Security, round 11 Blocker · 5 Major · 1 High · 5 Medium

Adversarial review of every checkpoint before code review even started — caught a repo-state bug, a cookie SameSite gap, a double-API-prefix bug, and no rate limiting, among others.

Fix round8/8 verified fixed

All round-1 findings fixed and independently re-verified against real code, not just re-read.

Code review loop2 iterations → PASS, ≪ $300 cap

System-wide review found unsafe param coercion and zero service/gateway/repo specs on iteration 1; both fixed; iteration 2 passed clean.

Live sandbox runpost-build bug hunt

Ran the whole stack end-to-end for real, then did a dedicated post-run bug hunt and refactor pass before calling core scope done.

Bonus 1 — AuthenticationHLD → design → build → QA/security

has_secure_password (bcrypt), signed HttpOnly identity cookie, username case-normalized both app- and DB-index-side, timing-safe login (a dummy bcrypt compare on a miss, to close a username-enumeration side channel).

Bonus 2 — Live deploymentRender + MongoDB Atlas

Switched from an initial Fly.io plan (card required, no real free tier) to Render (genuinely free, no card). Two independent services — Static Site + Docker Web Service — on different subdomains, so the auth cookie became a real cross-site cookie; the CROSS_ORIGIN_COOKIES flag designed in during Bonus 1, unused until now, is what made that a config flip instead of new work.

Bonus 3 — Twilio delivery-status webhookssignature-authenticated, live-verified

Inbound webhook authenticated by Twilio's own HMAC-SHA1 request signature — not the user cookie, since Twilio can't hold one. Verified against the live deployment by hand-building a genuinely valid signature with openssl and confirming a real message's status flipped sent → delivered.

Bug blitz, two passes1 High · 2 Medium · 3 Low, all closed

A systematic cross-cutting re-review after every bonus was live. Found and fixed: a message-history store never cleared on logout (cross-user leak risk on a shared browser); SMS content leaking into request logs; an unhandled 500 on a rare signup race; a stale/failing frontend test; render.yaml drift from the real deploy; an unbounded message-history query (now capped); a Twilio webhook with no ordering guard, which could have let a delayed callback regress a delivered message's status backward; and zero field validations on the Message model. Backend test suite grew from 48 to 52 passing runs along the way.

Closing the loop on the original prompt2026-07-16

Re-reading the original spec verbatim (to build this very infographic) surfaced that Bonus 3's literal ask — "a reflection to the message cards showing that twilio successfully delivered the message" — had been left as an open design question and never actually built on the frontend, even though the backend had supported it since Bonus 3 shipped. Fixed same-day: a color-coded delivery-status badge now appears on every message card.
