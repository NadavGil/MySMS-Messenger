# Bug Blitz — 2026-07-15

A systematic bug-hunting pass across the whole app (backend, frontend,
deployment config, test suites) run after all three bonus features were live
and verified. Unlike the earlier `qa-security-review-*` passes, this wasn't
scoped to one feature — it re-read the whole codebase looking for anything
new or previously missed. Every finding below was independently verified by
direct file inspection (and, where testable, by actually running the
relevant test suite) before being reported as real; suspected false
positives are called out explicitly.

## Frontend

### HIGH — message history leaked across logins on a shared browser (fixed)

`MessagesStoreService` is `providedIn: 'root'`, a singleton that outlives any
single user's session. Nothing cleared it on logout, so on a shared/kiosk
machine a user logging in right after someone else logged out would briefly
see the previous user's message count in the `<h2>Message History (N)</h2>`
header — that count isn't gated behind the `loading$`/`error$` conditional
the message list itself uses, so it stayed stale until the next `refresh()`
resolved.

**Fix:** added `MessagesStoreService.clear()` and wired it up in
`AppComponent.ngOnInit()`, which subscribes to `AuthStoreService.loggedIn$`
and clears the store whenever it goes `false` — covering both the explicit
logout button and the automatic 401 `clearSession()` path. Deliberately
**not** wired into `AuthStoreService` itself: doing so would create a
circular DI dependency (`AuthStoreService → MessagesStoreService →
MessagesApiService → AuthStoreService`, since the API layer already depends
on the auth store for its 401 handling) that Angular's DI would reject at
runtime (`NG0200`). Caught before it shipped by tracing the full dependency
chain; fixed at the component level instead.

### Stale regression test caught the drift, once actually run

`messages-api.service.spec.ts` still asserted `environment.production.ts`'s
`apiBaseUrl` was an empty string (the pre-Bonus-2 same-origin assumption).
Bonus 2 deliberately moved production to a genuine two-service, cross-origin
Render deploy with an absolute API origin, but this test was never updated
to match and had been failing silently because `ng test` wasn't re-run after
that config change landed. Updated the assertion to match the real
convention (non-empty absolute origin, no trailing slash, no double
`/api/api`) while keeping the original regression guard intact. Full suite
now passes: **74/74 tests, 11/11 files.**

### Verified false positive: signup username regex

A reviewing subagent flagged `signup.component.ts`'s
`Validators.pattern(/^[a-zA-Z0-9_]{3,30}$/)` as inconsistent with the backend
(which appeared to require lowercase only). Checked directly:
`backend/app/models/user.rb` downcase-normalizes usernames before validation,
and `auth_controller.rb#login` downcases on lookup — so the backend is
genuinely case-insensitive end to end, and the frontend allowing mixed-case
input is correct by design, not a bug. No change made.

## Backend

### MEDIUM — SMS body/phone number were leaking into Rails request logs (fixed)

`Gateways::FakeSmsGateway` deliberately never logs the raw message body, but
that discipline never reached the framework-level request logger:
`config/initializers/filter_parameter_logging.rb` didn't include `body` or
`to_number`, so `ActionController::LogSubscriber`'s default
`Parameters: {...}` log line printed the full SMS content on every send
regardless. Added `:body, :to_number` to `filter_parameters`.

### MEDIUM — signup race could surface a raw 500 instead of "username taken" (fixed)

`User#username` uniqueness is validated app-side but explicitly documented
as "racy; index is authoritative" — the DB-level unique index is the real
guard. Two concurrent signups for the same normalized username could both
pass the Mongoid check and race at `save`; the loser would raise a raw
`Mongo::Error` (E11000) that nothing caught (`ApplicationController` only
rescues `Repositories::RepositoryError`, which `User` — a plain Mongoid
model, not a repository — never raises). Added a `rescue Mongo::Error`
around `AuthController#signup` that responds with the same structured 422
`{"username": ["has already been taken"]}` a normal validation failure
gives, instead of an unhandled 500 with a driver backtrace.

### LOW / informational — flagged but not changed this pass

- **No pagination on `GET /api/v1/messages`.** Full owner-scoped history is
  returned unbounded in both repository implementations. Likely an accepted
  scope limit rather than a regression, but will degrade over time on a
  long-lived account.
- **Twilio status webhook applies callbacks with no ordering guard.** A
  delayed/retried callback (e.g. `sent` arriving after `delivered`) could in
  theory regress a message's status backward. Needs verification against
  Twilio's actual delivery/retry guarantees before treating as a real bug.
- **`MessageDocument` has no field validations.** Not currently exploitable
  (the only writer path already validates upstream), but if a validation is
  ever added later, `MongoMessageRepository#create!` only rescues
  `Mongo::Error`, not `Mongoid::Errors::Validation` — worth a note for future
  changes.

No auth-bypass, CORS wildcard, NoSQL-injection, or rate-limit-bypass issues
were found in this pass.

## Deployment / config consistency

`render.yaml` had drifted from the actual live deployment in two ways (it
was never applied via the Blueprint flow — the real services were created
manually, per the known Blueprint payment-info gotcha already documented):

1. Still used the original placeholder service names
   (`mysms-messenger-api`/`mysms-messenger-web`) while README,
   `environment.production.ts`, and `tech-design.md` had all long since moved
   to the real deployed names (`mysms-messenger-server`/
   `mysms-messenger-ehtu`).
2. Was missing all four Bonus 3 Twilio env vars
   (`TWILIO_ACCOUNT_SID`/`TWILIO_AUTH_TOKEN`/`TWILIO_FROM_NUMBER`/
   `TWILIO_STATUS_CALLBACK_URL`) entirely — it was written before Bonus 3
   existed and never backfilled, even though `.env.example` already
   documents all four and they're genuinely set on the live backend.

Added a drift note plus the four missing vars (as `sync: false` placeholders,
matching the existing `SECRET_KEY_BASE`/`MONGO_URI` pattern) so a future
from-scratch Blueprint deploy wouldn't silently boot with the status webhook
permanently disabled. Left the placeholder service names as-is with an
explanatory comment, since this file is intentionally kept as illustrative
IaC documentation rather than a live-synced config — the actual URLs it
would need to match live entirely in the Render dashboard.
